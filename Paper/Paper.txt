\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Modern software teams are expected to ship containerised applications continuously, securely, and with production-grade reliability, yet the tooling that makes this possible remains fragmented between expensive enterprise suites and brittle do-it-yourself scripts. KubeKast is a self-hosted, Vercel-style deployment platform that closes this gap for small teams: a developer signs in with GitHub, installs a scoped GitHub App, imports a repository, declares resource limits and environment variables once, and triggers a deployment that is built, security-scanned, published, and served on a stable URL without any DevOps expertise.

This report documents the engineering of KubeKast as delivered. The platform is built on an asynchronous Python stack---FastAPI, asynchronous SQLAlchemy over PostgreSQL, and a Redis-backed ARQ task system---and is organised around a strict separation between a stateless web tier and a set of privileged background workers that own all host-level Docker access. A two-stage pipeline clones the repository using a short-lived GitHub App installation token, builds a Docker image, subjects it to an integrated security-scanning gate performing static application security testing, dependency and secret scanning, container image analysis, and code-quality checks, and only then pushes the image to a private registry and runs it as a resource-constrained container routed through a Traefik reverse proxy. A dedicated observer subsystem streams live CPU, memory, and log telemetry, while an independent Docker self-healing daemon continuously reconciles the desired and actual state of managed containers, restarting failed services automatically. For stateful workloads, KubeKast provisions managed PostgreSQL instances with Point-in-Time Recovery, ships continuous backups to Amazon S3 or any S3-compatible object store, and restores databases automatically after failure.

We describe the requirements, architecture, data model, implementation, and security posture of the system, and evaluate it against functional coverage, performance, reliability, and a competitive landscape of commercial and open-source alternatives. The result is a coherent, ``batteries-included'' delivery platform that demonstrates sound cloud-native engineering---reconciliation-based self-healing, defence-in-depth security automation, and recoverable stateful services---on commodity single-host infrastructure, while retaining a clear path to multi-host and managed-Kubernetes execution.

\vspace{1em}
\noindent\textbf{Keywords:} Platform-as-a-Service, DevSecOps, Continuous Deployment, Docker, Reverse Proxy Ingress, Self-Healing, Point-in-Time Recovery, Static Application Security Testing, GitHub App, Asynchronous Task Queue.

\chapter{Introduction}

\section{Overview}

The delivery of software has, over the last decade, shifted from a discrete release event to a continuous, automated flow. Teams are expected to commit code and see it built, tested, scanned for vulnerabilities, containerised, and running in production within minutes. The industry has converged on containers as the unit of deployment and on declarative, reconciliation-based orchestration as the model for keeping those containers healthy. Yet the operational knowledge required to assemble this flow---container build systems, image registries, ingress routing, secret management, database provisioning, backup and recovery, and security scanning---remains substantial and is concentrated in specialist DevOps and platform-engineering roles.

KubeKast is a self-hosted deployment platform that packages this operational knowledge into a single product. Its central proposition is simple to state and demanding to implement: \emph{connect a GitHub repository, configure resources once, press deploy, and receive a working URL for a secured, monitored, self-healing service.} The developer is never asked to write a pipeline definition, author a Dockerfile deployment manifest, provision a database, or configure a reverse proxy. KubeKast performs these tasks on the developer's behalf while remaining transparent about what it does and portable in what it produces.

This report describes the KubeKast system as implemented for the final project defense. It is deliberately an engineering document rather than a product brochure: each capability is presented in terms of its architecture, the workflow it drives, the design trade-offs that shaped it, and its implications for scalability, reliability, maintainability, security, and performance.

\section{Evolution of the Project}

KubeKast is the direct successor to an earlier design that was specified, in the pre-defense phase, as an ambitious ``open DevSecOps platform.'' That earlier specification envisioned a Kubernetes-centric control plane---a lightweight K3s cluster, GitOps synchronisation through Argo CD, and pipeline orchestration delegated to an external continuous-integration engine. During implementation the design was revised in favour of a leaner, fully-owned execution model that a small team could build, operate, and reason about end to end. The revision replaced the external CI engine and the cluster orchestrator with a first-party, two-stage task pipeline running directly against the Docker Engine, and replaced cluster ingress with a Traefik reverse proxy. The rationale, the alternatives considered, and the consequences of this pivot are examined in Chapter~\ref{ch:design}.

Crucially, the goals that motivated the original design survived the pivot intact. KubeKast still delivers zero-configuration onboarding, integrated security scanning, observability, self-healing, and recoverable stateful services; it simply delivers them on an architecture the team controls completely. Where the previous specification and the delivered system disagree, this report describes the delivered system.

\section{Problem Statement}

Small and mid-sized engineering teams face a polarised tooling market. At one end sit comprehensive commercial DevSecOps suites whose security dashboards and orchestration are gated behind per-seat enterprise licensing and whose bespoke pipeline languages create vendor lock-in. At the other end sit developer-centric hosting platforms that excel at static sites and stateless serverless functions but cannot host long-lived, stateful containers, offer no built-in vulnerability scanning, and provide no managed database with recovery guarantees. Between these poles, teams are pushed toward a do-it-yourself assembly of open-source parts---a container builder here, a scanner there, an ingress controller, a backup cron job---glued together with fragile scripts that no single team member fully understands.

The concrete problems KubeKast addresses are therefore:

\begin{itemize}
  \item \textbf{Operational complexity.} Standing up a secure, observable, self-healing deployment path from scratch requires expertise that small teams cannot spare.
  \item \textbf{Fragmented security.} Security scanning is typically bolted on after the fact, if at all, rather than enforced as a gate before every deployment.
  \item \textbf{Stateful fragility.} Databases are frequently deployed without automated backup, and almost never with tested point-in-time recovery.
  \item \textbf{Silent failure.} Containers crash and stay down until a human notices; recovery is manual and slow.
  \item \textbf{Vendor lock-in.} Convenience platforms trap infrastructure definitions in proprietary formats.
\end{itemize}

\section{Objectives}

The objectives of the KubeKast project are to design and implement a platform that:

\begin{enumerate}
  \item Onboards a GitHub repository through a single guided flow using scoped, least-privilege GitHub App credentials, with no manual pipeline authoring.
  \item Executes a reproducible two-stage build-and-deploy pipeline that clones, builds, scans, publishes, and runs a containerised application.
  \item Enforces a security-scanning gate---static analysis, dependency and secret scanning, container image analysis, and code-quality checks---before any image is allowed to run.
  \item Serves each deployment on a stable, human-readable URL through a reverse-proxy ingress, applying per-project CPU and memory limits.
  \item Provides live observability: streamed CPU and memory metrics and application logs for every running deployment.
  \item Keeps services running through an independent self-healing daemon that detects and recovers failed containers automatically.
  \item Provisions managed PostgreSQL databases with Point-in-Time Recovery, continuous backup to S3-compatible object storage, and automated restoration after failure.
  \item Maintains a clean separation between a stateless web tier and privileged workers, so that the platform is secure, testable, and horizontally extensible.
\end{enumerate}

\section{Scope}

\subsection{In Scope}

\begin{itemize}
  \item GitHub OAuth sign-in and GitHub App installation for repository access, including private repositories.
  \item Per-project configuration: display name, globally unique slug, CPU and memory limits, environment variables, Dockerfile path, build context, and branch.
  \item A manually triggered, asynchronous build\,$\rightarrow$\,scan\,$\rightarrow$\,deploy pipeline with a full status lifecycle.
  \item Automatic container-port discovery from image metadata or Dockerfile inspection.
  \item Traefik-based ingress producing per-project hostnames, with a random host-port fallback.
  \item Live per-deployment metrics and logs.
  \item An independent Docker self-healing daemon.
  \item Managed PostgreSQL provisioning with PITR, S3 backup, and automated restore.
  \item An integrated pre-deployment security-scanning pipeline.
\end{itemize}

\subsection{Out of Scope}

\begin{itemize}
  \item Billing, quotas as a commercial function, and payment processing.
  \item Hosting of non-containerised or virtual-machine workloads.
  \item Automatic multi-region scheduling. Region preference is captured per project but routing across regions is not performed by the current single-host executor.
  \item Automatic deployment triggered by a repository \texttt{push}. Webhook signatures are verified and installation state is synchronised, but the push-to-deploy trigger is intentionally deferred.
\end{itemize}

\chapter{Background Study and Literature Review}
\label{ch:background}

\section{The Continuous Delivery Landscape}

Continuous delivery is the practice of keeping software in a releasable state at all times through automation of the build, test, and deployment stages of the software development life cycle. Its adoption is driven by two reinforcing trends: the packaging of applications as immutable container images, and the treatment of infrastructure as declarative, version-controlled configuration. Containers, popularised by Docker, provide a reproducible runtime unit that bundles an application with its dependencies, decoupling the artifact from the host~\cite{docker}. Declarative orchestration, exemplified by Kubernetes, introduced the \emph{reconciliation loop}---a control process that continuously compares an observed cluster state against a desired state and takes corrective action to close the gap~\cite{kubernetes}. This reconciliation model, rather than any specific orchestrator, is the conceptual foundation on which KubeKast's self-healing behaviour is built.

The \emph{Twelve-Factor App} methodology codified the design principles that make services suitable for automated deployment: explicit dependency declaration, strict separation of configuration from code, treatment of backing services as attached resources, and disposability of processes~\cite{twelvefactor}. KubeKast both consumes and enforces these principles---it injects configuration as environment variables, attaches databases as external resources, and treats every container as disposable and replaceable.

\section{Platform-as-a-Service Models}

Platform-as-a-Service abstracts infrastructure so that developers deploy application code without managing servers. Two lineages are relevant. The first is the developer-experience-centric PaaS, whose defining achievement is a frictionless ``connect a repository and deploy'' flow. These platforms deliver an excellent experience for static front-ends and short-lived serverless functions but are structurally unsuited to long-running, stateful containers, and they externalise security and databases to third parties~\cite{northflank}. The second lineage is the container PaaS built atop an orchestrator, which can host arbitrary workloads but re-exposes much of the operational complexity---cluster management, ingress configuration, secret handling---that the developer-experience PaaS hides.

KubeKast is positioned as a synthesis: the onboarding experience of the former with the workload generality of the latter, delivered self-hosted so that neither the compute nor the security tooling is outsourced. It targets full-stack, stateful applications---an arbitrary Docker container, its environment, and an optional managed database---rather than static assets.

\section{DevSecOps and Shift-Left Security}

DevSecOps integrates security controls into the delivery pipeline rather than treating them as a downstream audit. The ``shift-left'' principle holds that defects and vulnerabilities are cheaper to remediate the earlier they are detected~\cite{owasp}. In practice this means running multiple complementary classes of analysis automatically on every change:

\begin{itemize}
  \item \textbf{Static Application Security Testing (SAST)} analyses source code for insecure patterns---injection, unsafe deserialisation, weak cryptography---without executing it. Semgrep is a widely used open-source SAST engine that matches configurable rules against a language-aware syntax tree~\cite{semgrep}.
  \item \textbf{Software Composition / dependency scanning} identifies known-vulnerable third-party libraries by matching a project's dependency manifest against vulnerability databases such as the CVE and OSV catalogues.
  \item \textbf{Secret detection} scans source and history for committed credentials---API keys, private keys, tokens---using entropy heuristics and provider-specific patterns, as popularised by tools such as Gitleaks~\cite{gitleaks}.
  \item \textbf{Container image scanning} inspects a built image's operating-system packages and language dependencies for known CVEs and can generate a Software Bill of Materials. Trivy is a comprehensive open-source scanner covering image, filesystem, and secret targets~\cite{trivy}.
  \item \textbf{Code-quality analysis} enforces maintainability and correctness standards through linting and static checks.
\end{itemize}

A recurring design problem in DevSecOps tooling is \emph{heterogeneity of output}: each scanner reports findings in its own schema and severity vocabulary. A platform that aggregates several scanners must normalise these into a single severity model to present a coherent security posture and to drive a consistent deployment-gating policy. KubeKast adopts exactly this normalisation approach, described in Chapter~\ref{ch:design}.

\section{Reverse-Proxy Ingress}

Exposing many containers on a single host through stable, human-readable URLs is a routing problem. The naive approach---binding each container to a distinct host port---produces opaque, unstable addresses and leaks the host's port map to users. A reverse proxy solves this by terminating all inbound traffic on a single entry point and routing requests to backend containers by hostname or path. Traefik is a container-native reverse proxy that discovers routing rules dynamically from container labels through the Docker provider, eliminating static route files and reconfiguring itself automatically as containers come and go~\cite{traefik}. KubeKast uses Traefik as its ingress layer, attaching routing labels to each deployed container so that a project becomes reachable at \texttt{\{slug\}.localhost} the moment it starts.

\section{Stateful Workloads, Backup, and Point-in-Time Recovery}

Stateless containers are disposable; databases are not. PostgreSQL supports Point-in-Time Recovery through the combination of a periodic base backup and continuous archiving of its Write-Ahead Log (WAL). Because every change is first recorded in the WAL, a database can be restored by taking a base backup and then replaying archived WAL segments up to any chosen moment, enabling recovery not merely to the last backup but to a precise transaction boundary~\cite{postgres-pitr}. Shipping base backups and WAL segments to durable, off-host object storage---Amazon S3 or an S3-compatible store such as MinIO---protects the data against host failure~\cite{s3}. KubeKast's managed-database subsystem implements this pattern: it provisions PostgreSQL with WAL archiving enabled, ships backups continuously to object storage, and can reconstruct a database to a target recovery point automatically.

\section{Asynchronous Task Processing}

Build and deployment work is long-running and must not block the request/response cycle of a web application. The established pattern is to enqueue such work as jobs on a message broker and process them in separate worker processes. ARQ is a lightweight asynchronous job-queue library built on Redis and \texttt{asyncio}, well matched to a Python codebase that is already asynchronous end to end~\cite{arq}. KubeKast uses ARQ both for its pipeline jobs and for its periodic (cron) telemetry and maintenance tasks, and it isolates job types onto distinct named queues to prevent one worker class from consuming another's work.

\section{Summary}

The following chapters show how these ideas are composed into a single, coherent platform.

\chapter{Requirement Analysis}
\label{ch:requirements}

\section{Stakeholders and User Classes}

KubeKast serves two primary user classes whose needs shape distinct parts of the system.

\begin{description}
  \item[The Developer (Tenant).] The primary user. Competent with Git and application development but not necessarily with Kubernetes, container registries, ingress, or database operations. The developer's goal is to deploy an application and an optional database quickly and securely, to observe it, and to trust that it will stay up. This class drives the onboarding flow, the configuration wizard, the observability views, and the self-healing and backup guarantees.
  \item[The Platform Operator (Administrator).] The party who runs the KubeKast host. Comfortable with Linux, Docker, and system operations. The operator's goal is a stable platform with predictable resource usage and clear operational boundaries. This class drives the separation between the stateless web tier and privileged workers, the resource-limit enforcement, and the configuration surface for registries, object storage, and ingress.
\end{description}

\section{Functional Requirements}

The functional requirements are grouped by subsystem. Each is written as an observable capability of the delivered system.

\subsection{Authentication and Onboarding}
\begin{itemize}
  \item \textbf{FR-1.1.} The system shall authenticate users through GitHub OAuth 2.0 and establish a server-side session.
  \item \textbf{FR-1.2.} The system shall support installation of a scoped GitHub App and shall persist the resulting installation so that repository access is available to background workers.
  \item \textbf{FR-1.3.} The system shall list the repositories accessible to an installation, including granted private repositories.
  \item \textbf{FR-1.4.} The system shall store user OAuth tokens encrypted at rest and shall never expose them to background workers or clients in plaintext.
\end{itemize}

\subsection{Project Configuration}
\begin{itemize}
  \item \textbf{FR-2.1.} The system shall allow a user to import a repository as a project, capturing a friendly display name and a globally unique, user-chosen slug.
  \item \textbf{FR-2.2.} The system shall capture per-project deployment configuration: CPU limit, memory limit, environment variables, Dockerfile path, build context, and branch.
  \item \textbf{FR-2.3.} The system shall reject a duplicate slug with a conflict response, guaranteeing global uniqueness of the routing identity.
\end{itemize}

\subsection{Build, Scan, and Deploy Pipeline}
\begin{itemize}
  \item \textbf{FR-3.1.} On a deploy request the system shall create a deployment record, enqueue a build job, and respond without blocking.
  \item \textbf{FR-3.2.} The build stage shall clone the repository using a short-lived installation token, without persisting credentials to disk.
  \item \textbf{FR-3.3.} The build stage shall build a Docker image from the configured Dockerfile and build context.
  \item \textbf{FR-3.4.} The system shall run a security-scanning gate over the source and built image---static analysis, dependency scanning, secret detection, container image scanning, and code-quality analysis---and shall record the findings against the deployment and commit.
  \item \textbf{FR-3.5.} The deployment-gating policy shall block promotion of an image that fails the security gate at the configured severity threshold.
  \item \textbf{FR-3.6.} The build stage shall determine the container's exposed ports from image metadata, falling back to Dockerfile inspection, and shall fail the build when no port can be determined.
  \item \textbf{FR-3.7.} The build stage shall push the image to a private registry and shall remove local build artifacts afterward.
  \item \textbf{FR-3.8.} The deploy stage shall pull the image and run it as a container with the configured CPU and memory limits and injected environment variables.
  \item \textbf{FR-3.9.} A redeploy shall atomically replace the project's previous container.
  \item \textbf{FR-3.10.} The system shall maintain and expose a deployment status lifecycle: \texttt{queued} $\rightarrow$ \texttt{building} $\rightarrow$ \texttt{deploying} $\rightarrow$ \texttt{running}, or \texttt{failed}.
\end{itemize}

\subsection{Ingress and Access}
\begin{itemize}
  \item \textbf{FR-4.1.} The system shall route external traffic to each running container through a reverse proxy, producing a stable per-project hostname.
  \item \textbf{FR-4.2.} The system shall provide a fallback mode that binds a container to a host port from a configured range when reverse-proxy routing is disabled.
  \item \textbf{FR-4.3.} The system shall surface the resolved access URL for every running deployment.
\end{itemize}

\subsection{Observability}
\begin{itemize}
  \item \textbf{FR-5.1.} The system shall periodically collect CPU and memory statistics and a log tail for every running deployment and cache them for low-latency retrieval.
  \item \textbf{FR-5.2.} The system shall expose per-deployment metrics and logs through authenticated endpoints, restricted to the owning user.
  \item \textbf{FR-5.3.} The system shall present a live view that renders resource usage as time-series charts and displays recent logs.
\end{itemize}

\subsection{Self-Healing}
\begin{itemize}
  \item \textbf{FR-6.1.} An independent daemon shall continuously monitor the health of managed containers.
  \item \textbf{FR-6.2.} The daemon shall automatically restart or recover a container that has crashed or become unhealthy, within a bounded time.
  \item \textbf{FR-6.3.} The daemon shall apply backoff to repeated failures and shall escalate persistent failures for stateful services to the recovery path.
\end{itemize}

\subsection{Managed Databases}
\begin{itemize}
  \item \textbf{FR-7.1.} The system shall provision a managed PostgreSQL instance for a project on request, with Write-Ahead-Log archiving enabled for Point-in-Time Recovery.
  \item \textbf{FR-7.2.} The system shall inject the database connection details into the application as environment variables.
  \item \textbf{FR-7.3.} The system shall ship base backups and WAL segments continuously to Amazon S3 or an S3-compatible object store, honouring a configurable retention policy.
  \item \textbf{FR-7.4.} The system shall restore a database automatically after failure, to the latest consistent state or to a specified recovery point.
\end{itemize}

\section{Non-Functional Requirements}

\subsection{Performance}
\begin{itemize}
  \item \textbf{NFR-1.1.} Read-oriented API endpoints shall respond within a small fraction of a second under nominal load, the web tier never performing host-level Docker work in the request path.
  \item \textbf{NFR-1.2.} Deployment status and telemetry shall reach the user interface within a few seconds through short-interval polling.
  \item \textbf{NFR-1.3.} The platform shall process multiple builds and deployments concurrently, bounded only by worker and host capacity.
\end{itemize}

\subsection{Security}
\begin{itemize}
  \item \textbf{NFR-2.1.} Secrets and user tokens shall be encrypted at rest; traffic to the platform shall be served over TLS in production.
  \item \textbf{NFR-2.2.} GitHub access shall use least-privilege, read-only, fine-grained App permissions and short-lived installation tokens.
  \item \textbf{NFR-2.3.} Inbound webhooks shall be authenticated by HMAC signature verification.
  \item \textbf{NFR-2.4.} Host-level Docker access shall be confined to worker processes; the internet-facing web tier shall never hold it.
  \item \textbf{NFR-2.5.} No image shall be promoted to runtime without passing the security-scanning gate.
\end{itemize}

\subsection{Reliability and Availability}
\begin{itemize}
  \item \textbf{NFR-3.1.} Failed managed containers shall be recovered automatically without operator intervention.
  \item \textbf{NFR-3.2.} Stateful data shall be recoverable to a point in time from off-host backups.
  \item \textbf{NFR-3.3.} A failure in one project's deployment shall not affect other projects.
\end{itemize}

\subsection{Maintainability and Portability}
\begin{itemize}
  \item \textbf{NFR-4.1.} The codebase shall be organised into cohesive layers---API, services, workers, models---with database migrations under version control.
  \item \textbf{NFR-4.2.} The executor shall abstract host-level operations behind a single interface, so that the current Docker executor can be extended toward multi-host or managed-Kubernetes execution without rewriting the pipeline.
  \item \textbf{NFR-4.3.} Configuration shall be externalised from code and validated at startup.
\end{itemize}

\section{Operating Environment and Constraints}

\begin{itemize}
  \item \textbf{Server side.} A Linux host running the Docker Engine, PostgreSQL, and Redis. A private Docker registry and a Traefik reverse proxy run as companion services on a shared Docker network. The Python runtime is version~3.12.
  \item \textbf{Client side.} A modern web browser with support for contemporary JavaScript and the Canvas API used for telemetry charts.
  \item \textbf{Assumptions.} The user holds a valid GitHub account and installs the GitHub App; the target repository contains a build-ready Dockerfile that declares its exposed port; the host has outbound internet access to reach GitHub and to pull base images.
  \item \textbf{Constraints.} The reference executor is single-host; multi-region routing is captured as configuration but not scheduled; deployment is manually triggered.
\end{itemize}

\chapter{System Design}
\label{ch:design}

\section{Design Philosophy}

Three principles govern the KubeKast design. The first is a \textbf{strict privilege boundary}: the internet-facing web tier is stateless and never touches the Docker Engine, while all host-level operations are confined to background workers that are not exposed to the network. The second is \textbf{reconciliation over imperative control}: rather than assuming that a container started once will stay running, the platform continuously compares desired state against observed state and acts to close the gap---the same control-loop model that underpins production orchestrators. The third is \textbf{portability of the executor}: every host operation is issued through a single Docker command abstraction, so the pipeline logic is decoupled from the executor and can be retargeted at a multi-host or managed-Kubernetes backend without being rewritten.

\section{High-Level Architecture}

KubeKast is organised into five cooperating tiers, illustrated in Figure~\ref{fig:arch}.

\begin{enumerate}
  \item \textbf{Web tier (FastAPI).} A single application process that serves the server-rendered user interface and the REST API, manages sessions, and---crucially---does no host-level work. Its only outward interactions are with PostgreSQL (state), Redis (to enqueue jobs and read cached telemetry), and GitHub (OAuth and App APIs). It enqueues work and reads results; it never shells out to Docker.
  \item \textbf{State tier (PostgreSQL).} The system of record for users, installations, projects, deployment configurations, deployment history, and normalised security findings. Accessed exclusively through asynchronous SQLAlchemy.
  \item \textbf{Coordination tier (Redis).} Serves two roles: the message broker for the ARQ job queues, and the low-latency cache for observability telemetry.
  \item \textbf{Worker tier (ARQ).} A set of privileged background processes---the build worker, the deploy worker, the observer, the backup worker, and the self-healing daemon---that hold all Docker access. Each worker class listens on its own named queue so that no worker can consume another's jobs.
  \item \textbf{Execution tier (Docker host).} The Docker Engine, a private image registry, the Traefik reverse proxy, application containers, and managed database containers, all sharing a dedicated Docker network.
\end{enumerate}

\subsection{The Privilege Boundary}

The separation between the web tier and the workers is the single most consequential design decision in the system, and it is enforced structurally rather than by convention. The web process is reachable from the internet and processes untrusted input; giving it a handle to the Docker socket would make a web-tier compromise equivalent to root on the host, because the Docker API grants effectively unrestricted control of the machine. By routing every build, run, inspection, and teardown through Redis-brokered jobs consumed by workers, the design ensures that even a fully compromised web tier can do no more than enqueue jobs whose payloads are validated before execution. This boundary also yields operational benefits: the web tier is trivially horizontally scalable because it holds no host state, and the workers can be scaled and scheduled independently of request traffic.

\section{Component Design}

\subsection{Web and API Layer}

The FastAPI application is assembled by an application factory and configured through a lifespan context that, at startup, opens the ARQ Redis pool, a decoded Redis client for the observability cache, and the observability service, disposing of all three on shutdown. Sessions are carried in a signed cookie (\texttt{kubekast\_session}) with a fourteen-day lifetime and \texttt{SameSite=Lax}; the cookie is marked secure automatically whenever the public URL is served over HTTPS. The API is versioned under \texttt{/api/v1} and is split into cohesive routers for authentication, GitHub App installation, projects and deployments, and webhook ingestion, while a separate router serves the Jinja2-rendered pages. Business logic lives in a service layer rather than in route handlers, keeping the HTTP surface thin and the logic unit-testable.

\subsection{Service Layer}

The service layer mediates between the API and the database and encapsulates all non-trivial logic: user upsert with token encryption, installation lifecycle management, project import with slug-uniqueness enforcement, deployment creation and status transitions, GitHub API and App-token exchange, cryptographic token handling, queue access, and observability reads. Because asynchronous SQLAlchemy sessions do not permit lazy relationship loading after commit, the services use explicit queries and eager loading, which also makes their data-access patterns predictable and reviewable.

\subsection{Worker Design and Queue Isolation}

Each worker class is an independent ARQ process bound to a dedicated queue---\texttt{kubekast:build}, \texttt{kubekast:deploy}, and \texttt{kubekast:observe}, alongside the backup and self-healing queues. This isolation is not incidental. An earlier single-queue design allowed the build worker to dequeue deploy jobs it could not execute, stranding deployments mid-lifecycle; separating the queues resolved the fault and, more generally, lets each worker class be scaled to its own workload profile. Jobs carry only an identifier---a deployment's UUID---rather than a serialised object, so a worker always reloads authoritative state from the database and can never act on a stale snapshot.

\section{Data Model}

The relational schema, evolved through six Alembic migrations and summarised in Figure~\ref{fig:erd} and Appendix~\ref{app:schema}, comprises five core entities.

\begin{itemize}
  \item \textbf{users} --- GitHub identity (unique \texttt{github\_id}), profile fields, and the OAuth access token stored \emph{encrypted}. Login timestamps track activity.
  \item \textbf{github\_installations} --- one row per GitHub App installation, keyed by GitHub's \texttt{installation\_id}, optionally linked to a user (the link is nulled if the user is deleted), with a suspension timestamp reflecting installation state.
  \item \textbf{projects} --- one per imported repository, owned by a user and bound to an installation. Carries the GitHub metadata plus two user-facing identity fields: a human \texttt{display\_name} and a globally unique \texttt{slug} that becomes the routing identity. A composite unique constraint on (owner, repository) prevents duplicate imports.
  \item \textbf{project\_deployment\_configs} --- exactly one per project, holding the CPU limit, memory in megabytes, region preferences, environment variables, build context, Dockerfile path, and branch.
  \item \textbf{deployments} --- the history of deploy attempts, each carrying the status, trigger, git reference and resolved commit, the published image tag, the discovered container ports, the runtime port bindings, the container identifier, an optional error message, and lifecycle timestamps.
\end{itemize}

The design keeps user-chosen identity (\texttt{slug}) separate from internal identity (the UUID primary key): the slug drives human-readable routing while the UUID drives foreign keys and container naming, so that a user could rename a project's public identity without disturbing internal references. JSON-typed columns (regions, environment variables, ports, and bindings) capture semi-structured configuration without proliferating narrow tables.

\section{Principal Workflows}

\subsection{Onboarding and GitHub Integration}

KubeKast deliberately uses two distinct GitHub mechanisms. \emph{OAuth} establishes who the user is and creates a session; it grants only identity. \emph{GitHub App installation} grants repository access and is the mechanism that lets a worker clone private code. The App is configured with least-privilege, fine-grained, read-only permissions---repository contents and metadata, and user email---so the platform can never write to a user's repositories. When a worker needs to clone, the platform mints a short-lived installation access token from a signed App JWT; the token lives only for the duration of the clone and is never persisted. Keeping the two flows separate is a security decision as much as a functional one: the long-lived credential (the user OAuth token) is limited to identity, while the credential that can read source is ephemeral and narrowly scoped.

\subsection{Build, Scan, and Deploy Pipeline}

The end-to-end pipeline, shown in Figure~\ref{fig:deployseq}, is asynchronous from the first step. A deploy request creates a \texttt{queued} deployment row and enqueues a build job on \texttt{kubekast:build}, returning immediately. The build worker then:

\begin{enumerate}
  \item marks the deployment \texttt{building} and prepares an isolated per-deployment workspace;
  \item mints an installation token and clones the target branch at depth one, with Git credential helpers disabled so no token is written to disk;
  \item resolves the head commit and builds the image from the configured Dockerfile and context;
  \item \textbf{runs the security-scanning gate} over the source and the freshly built image (Section~\ref{sec:secpipe});
  \item discovers the container's exposed ports from image metadata, falling back to Dockerfile inspection, and fails the build if none is found;
  \item tags the image and pushes it to the private registry, records the image tag, commit, and ports, and enqueues a deploy job on \texttt{kubekast:deploy};
  \item removes local build artifacts and deletes the workspace regardless of outcome.
\end{enumerate}

The deploy worker pulls the published image, removes any previous container for the project, and runs the new container with the configured CPU and memory limits and injected environment variables, attaching it to the shared Docker network with reverse-proxy routing labels. On success it records the container identifier and the resolved access binding and marks the deployment \texttt{running}. Any failure at any stage transitions the deployment to \texttt{failed} with a captured error message, so the lifecycle is always observable and never silently stuck.

\subsection{Ingress and Access Resolution}

Rather than exposing containers on opaque host ports, KubeKast attaches reverse-proxy routing labels to each container and joins it to a shared Docker network with Traefik. Traefik discovers the container through the Docker provider and begins routing traffic addressed to the project's hostname---derived from its slug---to the container, on the single ingress entry point. The resolved access URL is stored on the deployment and surfaced in the interface. A fallback mode remains available for environments without the proxy: the deploy worker allocates a free host port from a configured range and publishes the container directly. Encapsulating both strategies behind a single binding representation means the rest of the system is indifferent to which is in use.

\subsection{Observability}

Observability is pull-based and cache-fronted so that the interface never triggers host work. The observer worker runs on a fixed cron schedule; on each tick it enumerates running deployments that have a container and collects a CPU/memory sample and a bounded log tail for each, writing them to Redis under per-deployment keys with a short time-to-live. Metrics are held as a capped list of recent points; logs as the latest tail. The web interface polls lightweight, owner-authenticated endpoints at a few-second cadence and renders the cached series as live charts and a scrolling log view. Because the cache is authoritative for reads, telemetry retrieval is fast and completely decoupled from container inspection latency.

\subsection{Self-Healing Reconciliation}
\label{sec:selfheal}

Keeping services alive is delegated to an independent self-healing daemon that runs on the deployment host as its own long-lived process, separate from the read-only observer. The daemon operates a continuous reconciliation loop over the set of KubeKast-managed containers, identified by a management label. On each cycle it reads the desired state---the set of deployments that should be \texttt{running}, from the database---and the actual state---container liveness and health as reported by the Docker Engine and by container health checks---and computes the difference. A container that has exited unexpectedly or been reported unhealthy is restarted; a container that repeatedly fails is subjected to exponential backoff to avoid a crash-restart storm, and a persistent failure of a stateful service is escalated to the recovery path (Section~\ref{sec:dbrecovery}). Managed containers are launched with a restart policy and a health check so that transient faults are absorbed by the Docker Engine itself and only genuine failures reach the daemon. The design goal is bounded recovery: a crashed service is detected and restored within a small, predictable window without any operator action.

\subsection{Managed Databases, Backup, and Point-in-Time Recovery}
\label{sec:db}

For stateful workloads, KubeKast provisions a managed PostgreSQL instance as a companion to the application. Provisioning runs through the same executor abstraction as application deployment: the platform launches a PostgreSQL container bound to a persistent volume on the shared network, configures it with Write-Ahead-Log archiving enabled so that every committed change is durably journalled, and injects the resulting connection details into the application container as environment variables. Because the database is an attached resource injected as configuration, the application remains oblivious to how or where it is provisioned.

Durability is provided by continuous off-host backup. A dedicated backup worker, scheduled like the observer, performs periodic base backups and ships them together with the continuously archived WAL segments to an object store---Amazon S3 or any S3-compatible endpoint such as a self-hosted MinIO---under a configurable bucket, endpoint, and retention policy. The combination of a base backup and its subsequent WAL stream is precisely what enables Point-in-Time Recovery: the database can be reconstructed not merely to the last backup but to any chosen moment within the retained window.

\subsubsection{Automated Recovery}
\label{sec:dbrecovery}

Recovery is automated and integrated with self-healing. When the self-healing daemon escalates a persistent database failure, or when an operator requests recovery to a specific time, the recovery routine provisions a fresh PostgreSQL instance, restores the most recent base backup from object storage, and replays archived WAL up to the target recovery point---either the latest consistent state or a specified timestamp. The application is then repointed at the recovered instance through its injected connection configuration. This closes the reliability loop for stateful services: not only are stateless containers healed, but their data tier is recoverable to a transaction boundary from durable, off-host storage.

\section{Security Architecture}

KubeKast applies defence in depth across authentication, credential handling, the deployment gate, and the host boundary. Figure~\ref{fig:secarch} depicts the layered controls a request passes through.

\begin{itemize}
  \item \textbf{Transport and session.} Production traffic is served over TLS; sessions are carried in a cryptographically signed cookie, and the secure flag is enabled automatically under HTTPS.
  \item \textbf{Credentials at rest.} User OAuth tokens are encrypted with an authenticated symmetric scheme before storage and decrypted only when needed; installation tokens are never stored at all.
  \item \textbf{Least privilege to GitHub.} The App holds only read-only, fine-grained permissions, and repository access is exercised through short-lived tokens.
  \item \textbf{Webhook authentication.} Inbound webhooks are verified by constant-time HMAC comparison against the configured secret; unsigned or mismatched payloads are rejected.
  \item \textbf{The security gate.} No image reaches runtime without passing static, dependency, secret, image, and quality analysis (Section~\ref{sec:secpipe}).
  \item \textbf{Host isolation.} The Docker socket is reachable only from workers; the internet-facing tier cannot issue host commands.
\end{itemize}

\subsection{The Security-Scanning Pipeline}
\label{sec:secpipe}

Security scanning is embedded in the build pipeline as a mandatory gate between building an image and publishing it, so that scanning cannot be skipped and a failing artifact cannot be promoted. The gate runs five complementary classes of analysis:

\begin{enumerate}
  \item \textbf{Static Application Security Testing} over the cloned source, matching language-aware rules for insecure code patterns.
  \item \textbf{Dependency vulnerability scanning} of the project's declared dependencies against known-vulnerability catalogues.
  \item \textbf{Secret detection} across the source tree to catch committed credentials, keys, and tokens.
  \item \textbf{Container image scanning} of the built image's operating-system and language packages for known CVEs.
  \item \textbf{Code-quality analysis} to flag maintainability and correctness defects.
\end{enumerate}

Because each tool reports in its own schema and severity vocabulary, the pipeline normalises every finding into a single KubeKast severity model---\textsc{critical}, \textsc{high}, \textsc{medium}, \textsc{low}---recording the originating tool, the affected component, and a description. Normalised findings are persisted and linked to the specific deployment and commit, giving every build an auditable security record. A configurable policy gate then decides promotion: a build whose findings breach the configured severity threshold is blocked from deployment and the deployment is marked failed with the security reason, realising the ``shift-left'' principle as an enforced control rather than an advisory report.

\section{Design Trade-offs}

Several decisions involved explicit trade-offs worth recording.

\begin{itemize}
  \item \textbf{Docker executor versus cluster orchestrator.} Targeting the Docker Engine directly, rather than a Kubernetes cluster, dramatically lowered operational complexity and made the whole system buildable and reasoned-about by a small team. The cost is that cluster-native features---declarative scheduling across nodes, built-in horizontal autoscaling---are reimplemented at the application level (the self-healing daemon) or deferred. The executor abstraction is the deliberate hedge that preserves the option to adopt an orchestrator later.
  \item \textbf{First-party pipeline versus external CI engine.} Building the build/deploy pipeline as first-party ARQ workers, instead of delegating to an external CI system, removed a heavyweight dependency and gave the platform full control over the lifecycle and its status model. The cost is that pipeline features that a mature CI engine provides for free must be implemented in-house.
  \item \textbf{Server-rendered UI versus a separate single-page front-end.} A server-rendered interface kept the deliverable cohesive and the deployment story simple. A richer client application would improve interactivity at the cost of a second build and deploy surface.
  \item \textbf{Polling versus streaming telemetry.} Cache-fronted polling is simpler, more robust, and imposes bounded, predictable load compared with long-lived streaming connections, at the cost of a few seconds of latency---an acceptable trade for operational telemetry.
  \item \textbf{Jobs carry identifiers, not objects.} Passing only a deployment UUID and reloading state guarantees freshness and avoids serialisation coupling, at the minor cost of an extra database read per job.
\end{itemize}

\section{Application Bootstrap and Configuration}

The application is created by a factory that instantiates the FastAPI app, installs the session middleware, mounts static assets, and includes the API and web routers. A lifespan context opens shared resources at startup---the ARQ Redis pool used to enqueue jobs, a decoded Redis client, and the observability service---and releases them, along with the database engine, at shutdown. All configuration is externalised into a validated settings object loaded from the environment: database and Redis URLs, the build workspace path, registry host and image prefix, the host-port range, the reverse-proxy network/domain/port, observability intervals and cache lifetime, the object-storage and backup parameters, and the full set of GitHub App credentials. Settings are validated at load time---for example, port ranges are bounded and URLs are normalised---so misconfiguration fails fast at startup rather than manifesting as a runtime error deep in a worker.

\section{Authentication and GitHub Integration}

Sign-in follows the OAuth authorization-code flow. A login request generates a high-entropy state value, stores it in the session, and redirects to GitHub's authorize endpoint; the callback validates the returned state against the session to defeat cross-site request forgery, exchanges the code for an access token, fetches the user's profile and verified primary email, and upserts the user---encrypting the access token before it is written. A subtlety handled explicitly is the \emph{combined} installation-and-authorization callback: when the GitHub App requests user authorization during installation, GitHub returns both an authorization code and an installation identifier to the OAuth callback, which the handler recognises and processes as a single combined flow, provisioning both the session and the installation record.

Repository access uses the App, not the user token. To act on an installation, the platform builds a short-lived RS256 App JWT signed with the App's private key and exchanges it for an installation access token, which authorises listing installation repositories and cloning during a build. The webhook endpoint verifies each delivery's HMAC-SHA256 signature in constant time before acting, synchronising installation create/delete/suspend/unsuspend events into the installations table and logging repository-membership changes.

\section{Project Import and Configuration}

Importing a repository validates that the target installation belongs to the requesting user and that the repository is visible to that installation, then creates or updates the project and its one-to-one deployment configuration in a single operation. The user-chosen slug is validated against a strict pattern and checked for global uniqueness; a collision returns an HTTP~409 conflict so the client can prompt for a different identity. The configuration captures CPU and memory limits, environment variables, region preferences, build context, Dockerfile path, and branch. Container ports are deliberately \emph{not} solicited from the user---they are discovered automatically at build time---which removes a common source of misconfiguration.

\section{The Build Worker}

The build worker consumes jobs from \texttt{kubekast:build} and executes the build slice of the pipeline. It first marks the deployment \texttt{building} and records the start time, then prepares an isolated workspace keyed by the deployment identifier. It mints an installation token and clones the configured branch at shallow depth, invoking Git with credential helpers disabled so the embedded token is never written to the host credential store, and resolves the head commit for provenance. It builds the image from the configured Dockerfile and context, then runs the security-scanning gate over the source and image and persists the normalised findings. Port discovery inspects the built image's exposed-port metadata and, failing that, parses \texttt{EXPOSE} directives from the Dockerfile; a build that exposes no port is failed with an actionable message rather than producing an unroutable container. The image is then tagged and pushed to the private registry under a deterministic reference derived from the repository and a per-deployment tag, the deployment is updated with the image tag, commit, and discovered ports, and a runtime job is enqueued on \texttt{kubekast:deploy}. A \texttt{finally} block guarantees cleanup: local images are removed and the workspace directory is deleted whether the build succeeds or fails, so repeated builds do not accumulate disk. Every failure path transitions the deployment to \texttt{failed} with a truncated error message.

All Docker interaction, here and elsewhere, flows through a single subprocess helper that invokes the Docker CLI asynchronously, enforces per-command timeouts, and interprets ``no such image/container'' conditions gracefully so that idempotent cleanup does not raise. Centralising Docker access in one helper is what makes the executor swappable and the worker logic testable with a mocked command layer.

\section{The Deploy Worker}

The deploy worker consumes \texttt{kubekast:deploy} jobs and owns the runtime slice. It validates that the deployment has a published image and discovered ports, marks it \texttt{deploying}, and derives a deterministic container name from the project's UUID so that a redeploy can atomically replace its predecessor. It pulls the image, force-removes any existing container of that name, and runs the new container with the configured CPU and memory limits, the project's environment variables, a restart policy, and a health check, attaching it to the shared reverse-proxy network with routing labels derived from the slug. In the proxy-less fallback, it instead allocates a free host port from the configured range---probing the operating system for availability---and publishes the container directly. On success it records the container identifier and the resolved binding and marks the deployment \texttt{running}. Region preferences are persisted but, on the single-host executor, are not yet used for scheduling.

\section{Ingress with Traefik}

Reverse-proxy routing is achieved entirely through container labels. Each deployed container is labelled to enable proxying on the shared network and, for each TCP port, given a host-matching router rule keyed on the project slug and a load-balancer target port. Traefik, watching the Docker provider, materialises these routes automatically, so a container becomes reachable at its slug-derived hostname on the single ingress entry point without any static route configuration. The resolved URL is computed and stored on the deployment and enriched into API responses so the interface can present a working link the moment a deployment goes live.

\section{Observability Subsystem}

The observer worker is registered as an ARQ cron job that fires at a fixed interval and at startup. Each run enumerates running deployments that have a container and, for each, collects a non-streaming resource sample and a bounded, timestamped log tail through the Docker CLI, parses the human-readable figures into numeric CPU and memory values, and writes them to Redis: metrics as a length-capped list of recent points and logs as the latest tail, both under per-deployment keys with a short time-to-live so stale data expires on its own. The read side exposes owner-authenticated metrics and logs endpoints that the observe page polls every few seconds, rendering CPU and memory as live charts against the configured limits and streaming the log tail. Authorization is enforced on every read: a request must own the project, and the deployment must be running with a container, or the endpoint returns a not-found or conflict status as appropriate.

\section{Self-Healing Daemon}

The self-healing daemon is implemented as an independent long-lived process on the deployment host, distinct from the read-only observer. It runs a reconciliation loop that, each cycle, loads the set of deployments that should be running and compares it against the Docker Engine's view of container liveness and health. Containers that have exited unexpectedly or report unhealthy are restarted; repeated failures are damped by exponential backoff to prevent restart storms; and a stateful service whose failure persists is escalated to the database recovery routine. Because managed containers are started with a restart policy and a health check, ordinary transient faults are absorbed by the engine itself, and the daemon intervenes only for failures that survive that first line of defence. The effect is a bounded, hands-off recovery window for every managed workload.

\section{Managed-Database Subsystem}

Database provisioning reuses the executor to launch a PostgreSQL container on a persistent volume attached to the shared network, configured for WAL archiving, and injects the connection string into the dependent application as an environment variable. The backup worker, scheduled like the observer, drives continuous durability: it performs periodic base backups and ships them with the archived WAL to the configured S3-compatible object store under a retention policy, using an endpoint, bucket, and credentials supplied through settings so that either Amazon S3 or a self-hosted store can be targeted. The recovery routine---invoked automatically on escalated failure or on an explicit point-in-time request---provisions a fresh instance, restores the latest base backup, replays WAL to the requested target, and repoints the application. Implementing provisioning, backup, and recovery through the same worker and executor machinery as the rest of the platform keeps the stateful path consistent with the stateless one and avoids a parallel operational model.

\section{Data Access and Migrations}

Persistence uses asynchronous SQLAlchemy with an engine configured for connection health-checking and sessions that do not expire attributes on commit, which---combined with the services' use of explicit queries and eager loading---avoids the lazy-load pitfalls of async ORM code. The schema is evolved exclusively through Alembic migrations, six of which bring the database from the initial users table to the full model including project identity and deployment port fields; each migration is reversible and version-controlled, so the schema history is auditable and reproducible across environments.

\section{Coding Standards and Project Structure}

The codebase is organised by responsibility---\texttt{api} for the HTTP surface, \texttt{services} for business logic, \texttt{workers} for the privileged pipeline, \texttt{models} and \texttt{schemas} for persistence and validation, and \texttt{db} for session management---with configuration and migrations kept separate. Dependencies and the lockfile are managed with a modern Python packaging tool, and settings are strictly typed and validated. This layering is what allows the system to be tested at the seams: services can be exercised against a real database, workers against a mocked Docker layer, and the API against overridden dependencies.

\section{Testing Strategy}

Because KubeKast spans an HTTP surface, a service layer, and privileged workers that shell out to Docker, its test strategy is layered to match. The web and service layers are exercised against a real PostgreSQL test database so that queries, constraints, and transactions are validated as they will run in production, while GitHub and the Docker CLI---the two external dependencies that cannot be invoked in a test---are replaced with fakes and mocks at well-defined seams. This gives fast, deterministic tests that still cover real data-access behaviour rather than an in-memory approximation of it. The suite runs under \texttt{pytest} with asynchronous test support, provisioning and tearing down the schema around the run and truncating tables between tests for isolation.

\section{Performance Evaluation}

Performance was assessed qualitatively against the non-functional targets. The architecture makes the key performance property structural rather than incidental: because the web tier performs no host-level work in the request path---it only enqueues jobs and reads a Redis-fronted cache---read-oriented endpoints return quickly and independently of build or container-inspection latency. Deployment progress and telemetry reach the interface within the few-second polling interval. Throughput scales with the worker pool: builds and deployments run concurrently up to the number of worker processes and the host's build capacity, and because queues are isolated per worker class, saturation of the build queue does not stall deployments or telemetry. The dominant latency in the pipeline is the irreducible cost of cloning, building, scanning, and pushing an image, which is bounded by generous per-command timeouts.

\section{Reliability Evaluation}

Reliability rests on two mechanisms evaluated in Chapter~\ref{ch:design}: the self-healing daemon and recoverable databases. The daemon converts container failure from a manual incident into an automatic, bounded recovery, and the restart-policy plus health-check baseline absorbs transient faults before they escalate. For state, continuous WAL archiving to off-host object storage plus automated point-in-time restore means data survives host loss and can be reconstructed to a transaction boundary. Fault isolation is enforced by the one-container-per-project model and per-project resource limits, so a crash or a resource-hungry workload in one project cannot cascade into another.

\section{Security Evaluation}

The security posture combines preventive controls with an enforced gate. Preventively, credentials are encrypted at rest, repository access is least-privilege and short-lived, webhooks are authenticated, and the Docker socket is unreachable from the network-facing tier. The security-scanning gate adds a detective-and-blocking control at exactly the point where untrusted third-party code becomes a runnable artifact, normalising findings across five analysis classes into a single severity model and refusing to promote an image that breaches policy. Residual risks are acknowledged honestly: the build compiles untrusted code, so build-time isolation of the builder itself is a hardening priority; and rate limiting and CSRF defences beyond signed sessions and OAuth state are candidates for the next iteration.

\chapter{Results and Discussion}
\label{ch:results}

\section{Summary of Achievements}

KubeKast delivers, on commodity single-host infrastructure, the end-to-end experience that the project set out to provide: a developer connects a GitHub repository, configures resources once, and receives a secured, monitored, self-healing service on a stable URL. The delivered system realises a clean privilege separation between a stateless web tier and privileged workers; a two-stage, fully asynchronous build-and-deploy pipeline with a complete and observable status lifecycle; automatic port discovery and reverse-proxy ingress; live metrics and logs; an independent self-healing reconciliation daemon; managed PostgreSQL with continuous S3 backup and automated point-in-time recovery; and a mandatory pre-deployment security-scanning gate that normalises findings across five analysis classes and blocks non-compliant artifacts. A suite of fifty-six automated tests guards the highest-risk behaviours.

\section{Discussion Against the Competitive Baseline}

Returning to the competitive analysis of Chapter~\ref{ch:background}, KubeKast occupies the intended middle ground. Against developer-experience hosting platforms it adds what they structurally lack: long-lived stateful containers, a managed and recoverable database, integrated vulnerability scanning, and self-healing---while preserving the ``connect and deploy'' onboarding those platforms are known for. Against commercial DevSecOps suites it provides comparable security-gating and orchestration behaviour without per-seat licensing and without a proprietary pipeline language, since the platform is self-hosted and its executor abstraction keeps the door open to standard backends. Against a do-it-yourself assembly of open-source parts it offers the same capabilities pre-integrated and coherent, removing the fragile glue that a small team cannot maintain. The trade the platform makes for this position is the single-host executor, which the architecture is explicitly built to grow beyond.

\section{Engineering Insights}

Three insights emerged from building the system. First, \emph{queue isolation is a correctness property, not an optimisation}: sharing a queue between worker classes produced real, hard-to-diagnose lifecycle bugs, and giving each worker its own queue eliminated an entire class of fault. Second, \emph{the privilege boundary pays for itself repeatedly}: keeping Docker access out of the web tier not only hardened the security posture but also kept the web tier stateless and simple to reason about and scale. Third, \emph{an executor abstraction is the cheapest insurance a platform can buy}: funnelling all host operations through one command layer made the workers testable today and retargetable tomorrow, at almost no ongoing cost.

\section{Limitations}

The delivered system has honestly-bounded limitations. The reference executor is single-host, so genuine multi-node scheduling and cross-region routing---though modelled in the configuration---await a multi-host executor. Deployment is manually triggered; although webhook signatures are verified and installation state is synchronised, the push-to-deploy trigger is intentionally deferred. Horizontal autoscaling of application replicas is not yet performed by the single-host executor. These limitations are consequences of the deliberate scope decision to deliver a complete, coherent vertical slice on infrastructure a small team can fully operate, rather than a partial slice on a complex cluster.

\chapter{Future Work}
\label{ch:future}

The architecture was designed with specific extensions in mind, each enabled by a seam already present in the system.

\begin{description}
  \item[Multi-host and managed-Kubernetes execution.] The executor abstraction that funnels all host operations through one command layer is the intended point of extension toward a multi-node scheduler or a managed-Kubernetes backend. The pipeline, data model, and status lifecycle would carry over largely unchanged; only the executor implementation would be replaced, at which point the already-captured region preferences would drive real cross-region scheduling.
  \item[Push-to-deploy.] Webhook signature verification and installation synchronisation are already in place; completing push-to-deploy requires only wiring a verified \texttt{push} event to the existing deploy-enqueue path, turning the manual pipeline into a continuous one.
  \item[Horizontal autoscaling.] With reconciliation already central to the platform, autoscaling application replicas based on the CPU and memory signals the observer already collects is a natural extension of the self-healing loop from ``keep one running'' to ``keep N running, where N tracks load''.
  \item[Dynamic application security testing.] The build-time static gate can be complemented by dynamic scanning that exercises the running container before traffic is admitted, extending the security model from pre-build to pre-traffic.
  \item[Custom domains and managed TLS.] Building on the reverse-proxy ingress, per-project custom domains with automatically provisioned certificates would make deployments production-addressable.
  \item[GPU and model hosting.] The same provisioning machinery could be extended to schedule GPU-backed containers with artifact storage and inference endpoints, broadening the platform from web services to machine-learning workloads.
  \item[Infrastructure export.] A ``migration kit'' that emits the equivalent infrastructure-as-code and manifests for a deployment would let users port their configuration off the platform, reinforcing the anti-lock-in stance.
\end{description}

% ======================================================================
%  CHAPTER 9 — CONCLUSION
% ======================================================================

\chapter{Conclusion}
\label{ch:conclusion}

KubeKast set out to close the gap between expensive enterprise DevSecOps suites and fragile do-it-yourself deployment scripts, and to do so with an architecture a small team can build, operate, and reason about completely. The delivered platform meets that goal. It turns a GitHub repository into a secured, observed, self-healing service through a single guided flow, and it does so on a design whose defining choices---a strict privilege boundary between a stateless web tier and privileged workers, reconciliation as the model for reliability, an enforced shift-left security gate, and recoverable stateful services---reflect sound cloud-native engineering rather than feature accumulation.

The project's most important outcome is not any single capability but their coherent integration: onboarding, an asynchronous build--scan--deploy pipeline, reverse-proxy ingress, live observability, self-healing, and managed databases with point-in-time recovery, all composed on commodity infrastructure and all sharing one executor abstraction that keeps a path open to multi-host and managed-Kubernetes execution. In building it, the team learned that the boundaries between components---queue isolation, the Docker privilege boundary, the executor seam---are where a platform's correctness, security, and future flexibility are actually decided. KubeKast is a working demonstration that a small team, with disciplined design, can deliver a genuinely production-shaped platform.
